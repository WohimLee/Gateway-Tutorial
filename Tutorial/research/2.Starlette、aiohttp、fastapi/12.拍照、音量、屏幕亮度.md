这类操作本质是**设备控制指令（control plane）**，特点很明确：

* 数据量小（几十字节级）
* 需要低延迟（但不是毫秒级硬实时）
* 需要可靠送达（不能丢）
* 可能需要状态确认（ack）
* 可能离线（设备不一定一直在线）

👉 所以它们和音视频完全是两类问题。

---

# 一、最适合的方案（直接结论）

## ✅ 优先选择：WebSocket 或 MQTT

| 场景                      | 推荐            |
| ----------------------- | ------------- |
| 设备长期在线（眼镜/手表实时连接）       | **WebSocket** |
| 设备可能离线 / 需要消息缓存 / IoT风格 | **MQTT**      |

---

# 二、为什么不用 WebRTC / HTTP？

### ❌ WebRTC 不适合

* 为音视频设计
* 建链复杂（ICE/STUN/TURN）
* 不适合简单控制指令

👉 用它发“拍照”这种指令 = 用导弹打蚊子

---

### ❌ HTTP 也不理想（单独用）

```text
POST /take_photo
POST /set_volume
```

问题：

* 设备必须轮询（延迟高）
* 无法主动推送
* 实时性差

👉 可以作为 fallback，但不适合主通道

---

# 三、WebSocket vs MQTT 怎么选？

## 方案一：WebSocket（推荐你们优先）

```text
设备 ←→ 网关（Starlette/FastAPI）
```

### 优点

* 实现简单
* 延迟低
* 双向通信天然支持
* 和你们 AI/token streaming 共用一条连接
* Android 实现容易

### 示例

```json
{
  "type": "command",
  "action": "take_photo",
  "request_id": "123"
}
```

设备返回：

```json
{
  "type": "ack",
  "request_id": "123",
  "status": "ok"
}
```

👉 这就是你们最典型的控制链路

---

## 方案二：MQTT（偏 IoT 架构）

```text
设备 ←→ MQTT Broker ←→ 后端
```

### 优点

* QoS（可靠投递）
* 离线消息
* 低功耗（适合手表/指环）
* 天然 pub/sub

### 示例 topic

```text
device/{id}/command
device/{id}/status
```

### 缺点

* 需要引入 broker（EMQX / Mosquitto）
* 体系更重
* 和 HTTP/AI 网关分裂（两套系统）

---

# 四、你们设备类型的建议

你们有：

* 智能眼镜（高性能）
* 手表（中等）
* 指环（低功耗）

👉 可以这样分：

## 智能眼镜（强设备）

👉 **WebSocket 优先**

原因：

* 本来就要跑 AI / token streaming
* 网络稳定（通常连手机）
* 可以一直在线

---

## 手表

👉 **WebSocket 或 MQTT 都可以**

看：

* 是否常在线 → WebSocket
* 是否省电优先 → MQTT

---

## 指环（低功耗）

👉 **MQTT 更合适**

原因：

* 省电
* 支持离线
* 网络不稳定

---

# 五、推荐你们的整体控制架构

```text
        ┌───────────────┐
        │ Starlette 网关 │
        │ (WebSocket)   │
        └──────┬────────┘
               │
       ┌───────┼────────┐
       │       │        │
   眼镜 WS   手表 WS   指环 MQTT
```

或者更统一一点：

```text
控制面：
  WebSocket（主）
  MQTT（低功耗设备）

数据面：
  WebRTC / HTTP / gRPC

业务/API：
  FastAPI / Starlette
```

# 六、一个容易踩坑的点（很关键）

👉 **一定要做“命令确认（ACK）机制”**

不要只发：

```json
{"action": "take_photo"}
```

而要：

```json
{
  "action": "take_photo",
  "request_id": "uuid"
}
```

然后设备必须回：

```json
{
  "request_id": "uuid",
  "status": "done"
}
```

否则你会遇到：

* 指令丢失
* 重复执行
* 状态不同步

---

# 七、一句话总结

> **拍照、音量、亮度这种控制指令：优先 WebSocket；低功耗设备用 MQTT；不要用 WebRTC，也不要只靠 HTTP。**
