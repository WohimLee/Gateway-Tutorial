## WebRTC 用什么实现？

| 位置          | 常用实现                                          |
| ----------- | --------------------------------------------- |
| Android 设备端 | **libwebrtc / Google WebRTC Native SDK**      |
| iOS 端       | libwebrtc Objective-C/Swift binding           |
| Web 端       | 浏览器原生 WebRTC API                              |
| Python 端    | **aiortc**                                    |
| Go/Rust 服务端 | Pion WebRTC / WebRTC.rs                       |
| 媒体服务器/SFU   | **LiveKit / mediasoup / Janus / Jitsi / SRS** |

Python 里最常见的是 **aiortc**，它是基于 asyncio 的 WebRTC/ORTC 实现。适合做原型、信令调试、抽帧分析、AI 处理接入，但不太建议拿 Python 自己硬扛大规模实时音视频转发。([aiortc][1])

你们这种智能眼镜场景，我建议：

```text
Android 眼镜
  ↓ WebRTC
LiveKit / Janus / mediasoup / SRS
  ↓ 抽音频/抽帧
AI 服务
```

Python 更适合做：

```text
FastAPI/Starlette 信令服务
aiortc 小规模处理/原型
AI pipeline 消费音频/图像帧
```

## MQTT 用什么实现？

| 位置          | 常用实现                                      |
| ----------- | ----------------------------------------- |
| Broker 服务端  | **EMQX / Mosquitto / HiveMQ / VerneMQ**   |
| Python 客户端  | **paho-mqtt / gmqtt / aiomqtt**           |
| Android 客户端 | Eclipse Paho Android / HiveMQ MQTT Client |
| 边缘设备        | Mosquitto client / embedded MQTT SDK      |

Python 最稳的是 **Eclipse Paho MQTT**，支持 MQTT 5.0、3.1.1、3.1。异步项目里可以看 **gmqtt / aiomqtt**。([Eclipse][2])

## 你们场景的推荐

```text
指令控制 / 状态 / 心跳：
MQTT 或 WebSocket

实时音频 / 视频：
WebRTC

业务 API / 设备注册 / 鉴权：
FastAPI 或 Starlette

AI token / 文本返回：
WebSocket 或 SSE
```

我的倾向：

```text
控制面：MQTT + Starlette/FastAPI
实时媒体面：WebRTC + LiveKit/Janus/SRS
业务网关：Starlette/FastAPI
Python WebRTC：只做原型或 AI 抽帧处理，不做核心 SFU
```

一句话：**MQTT 管设备控制，WebRTC 管实时音视频，Python/ASGI 管信令、鉴权、业务和 AI 编排。**
