**TypedDict**：适合描述“已经是 dict 的数据结构”。
常用于内部类型标注、HTTP JSON 响应/请求的静态提示。

```python
class UserPayload(TypedDict):
    id: int
    name: str
```

优点：轻量、无运行时成本。
缺点：不校验数据，运行时还是普通 dict。

---

**dataclass**：适合领域模型、内部业务对象。

```python
@dataclass
class User:
    id: int
    name: str
```

优点：简单、可读、适合业务逻辑。
缺点：默认不做强校验；从 JSON/dict 转换要自己处理或用额外库。

---

**Pydantic**：适合边界层数据校验和序列化，尤其是 HTTP/WebSocket 输入输出。

```python
class UserIn(BaseModel):
    id: int
    name: str
```

优点：自动校验、类型转换、错误信息清晰、JSON schema 支持好。
缺点：比 TypedDict/dataclass 重一些。

---

在 **HTTP API** 场景：

```text
外部请求 JSON -> Pydantic 校验 -> dataclass/领域模型处理 -> Pydantic/TypedDict 输出
```

推荐：

| 场景                  | 推荐                  |
| ------------------- | ------------------- |
| FastAPI 请求体/响应体     | Pydantic            |
| Flask/Django 手写校验   | Pydantic            |
| 内部业务对象              | dataclass           |
| 已知 JSON shape 的类型提示 | TypedDict           |
| 对性能极敏感、只要类型提示       | TypedDict/dataclass |

---

在 **WebSocket** 场景：

如果消息来自客户端，推荐 **Pydantic**：

```python
class ChatMessage(BaseModel):
    type: Literal["chat"]
    room_id: str
    text: str
```

因为 WebSocket 消息也是“不可信外部输入”，需要校验 `type`、字段、格式。

如果是内部广播消息，且结构稳定，可以用 `TypedDict` 或 `dataclass`。

---

我的经验性建议：

```text
外部边界：Pydantic
内部业务：dataclass
纯 dict 类型标注：TypedDict
```

HTTP/WebSocket 都属于外部边界，所以入口处优先用 **Pydantic**。
