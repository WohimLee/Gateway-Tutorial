我看了一圈 openclaw 的 gateway，核心结论是：

- HTTP 是一组 REST/SSE/OpenAI-compatible/静态资源接口
- WebSocket 是主控制面，统一 `req -> res` RPC，加 `event` 推送
- `res` 帧没有很多“种类”，只有统一 envelope；差异在 `method` 对应的 `payload`

**WebSocket 帧**
WS 顶层帧只有三类。

客户端请求：

```json
{
  "type": "req",
  "id": "request-id",
  "method": "chat.send",
  "params": {}
}
```

服务端响应：

```json
{
  "type": "res",
  "id": "request-id",
  "ok": true,
  "payload": {}
}
```

错误响应：

```json
{
  "type": "res",
  "id": "request-id",
  "ok": false,
  "error": {
    "code": "INVALID_REQUEST",
    "message": "..."
  }
}
```

服务端事件：

```json
{
  "type": "event",
  "event": "chat",
  "payload": {},
  "seq": 123,
  "stateVersion": {
    "presence": 1,
    "health": 2
  }
}
```

连接后第一帧是服务端发：

```text
event: connect.challenge
```

客户端必须先发：

```text
req method: connect
```

成功后服务端回：

```text
res payload.type: hello-ok
```

一些关键 event payload 形状：

```text
tick:
{ ts }

connect.challenge:
{ nonce, ts }

chat:
{ runId, sessionKey, seq, state, message?, errorMessage?, usage?, stopReason? }

agent:
{ runId, seq, stream, ts, data, sessionKey? }

node.invoke.request:
{ id, nodeId, command, paramsJSON?, timeoutMs?, idempotencyKey? }

talk.mode:
{ enabled, phase, ts }

voicewake.changed:
{ triggers }

exec.approval.requested:
{ id, request, createdAtMs, expiresAtMs }

exec.approval.resolved:
{ id, decision, resolvedBy, ts, request? }

device.pair.requested:
{ requestId, deviceId, publicKey, displayName?, platform?, ... }

device.pair.resolved:
{ requestId, deviceId, decision, ts }

node.pair.requested:
node pairing request object

node.pair.resolved:
{ requestId, nodeId, decision, ts }

session.message:
{ sessionKey, message, messageId?, messageSeq?, session?... }

sessions.changed:
{ sessionKey, phase, ts, messageId?, messageSeq?, session?... }

health:
health snapshot

presence:
presence entries / snapshot version related

update.available:
{ updateAvailable }
```
