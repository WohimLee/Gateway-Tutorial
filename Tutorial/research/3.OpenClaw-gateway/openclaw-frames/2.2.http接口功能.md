**REST 是什么**
REST 可以先理解成一种“用 HTTP 方法操作资源”的接口风格。

比如：

```text
GET    读取资源
POST   创建资源 / 执行动作
PUT    整体替换资源
PATCH  局部修改资源
DELETE 删除资源
```

REST 的核心不是某个框架，而是一种约定：URL 表示资源，HTTP 方法表示你要对资源做什么。

例如：

```text
GET /sessions/main/history
```

意思接近：

```text
读取 main 这个 session 的 history 资源
```

而：

```text
POST /sessions/main/kill
```

严格 REST 里可能更像“对 session 执行 kill 动作”，它不是特别纯的 REST 资源建模，但现实项目里这种 **REST-like action endpoint** 很常见。

**OpenClaw 里的接口说明**
健康检查 / 探活：

```text
GET/HEAD /health
GET/HEAD /healthz
```

用于判断 gateway 进程是否活着。一般给本地脚本、Docker、监控、负载均衡用。返回类似 `{ ok: true, status: "live" }`。

```text
GET/HEAD /ready
GET/HEAD /readyz
```

用于判断 gateway 是否“准备好服务”。`health` 只表示进程活着，`ready` 更偏向依赖、初始化、内部状态是否可用。没 ready 时可能返回 `503`。

OpenAI-compatible：

```text
GET /v1/models
GET /v1/models/{modelId}
```

模拟 OpenAI 的 models API。用于列出 OpenClaw 暴露出来的模型 ID，比如 `openclaw`、`openclaw/main`、`openclaw/{agentId}`。

```text
POST /v1/chat/completions
```

模拟 OpenAI Chat Completions API。外部程序可以像调用 OpenAI 一样调用 OpenClaw gateway，本质上会转成 OpenClaw 的 agent/chat 请求。

```text
POST /v1/responses
```

模拟 OpenAI Responses API。用途类似 `/v1/chat/completions`，但请求/响应格式走 Responses API 风格。

```text
POST /v1/embeddings
```

模拟 OpenAI Embeddings API。用于生成 embedding，背后会走 OpenClaw 配置的 memory embedding provider。

工具 / 会话：

```text
POST /tools/invoke
```

HTTP 方式直接调用 OpenClaw 工具。比如外部系统不走 WebSocket，也可以通过这个接口请求 gateway 执行某个 tool。

```text
POST /sessions/{sessionKey}/kill
```

终止某个 session 对应的运行，尤其是 subagent 或受控会话。`sessionKey` 是会话标识。

```text
GET /sessions/{sessionKey}/history
```

读取某个 session 的历史消息。默认返回普通 JSON。

如果带：

```text
Accept: text/event-stream
```

它会变成 SSE 流式接口，持续推送历史和新消息：

```text
event: history
event: message
```

Hooks / Webhook：

```text
POST {hooks.basePath}/wake
```

外部 webhook 唤醒 OpenClaw。比如收到某个自动化事件后，让 gateway 在下一次 heartbeat 或立即触发处理。

```text
POST {hooks.basePath}/agent
```

外部 webhook 直接触发一个 agent 请求。可以把外部系统的消息转成 OpenClaw agent turn。

```text
POST {hooks.basePath}/{mappingPath}
```

配置化 hook mapping。也就是不同路径可以按配置映射成 wake、agent request、指定 channel/session 等动作。

Canvas / A2UI：

```text
GET/HEAD /__openclaw__/a2ui...
```

返回 A2UI 前端资源。A2UI 可以理解成 OpenClaw 的一种辅助 UI/画布 UI 页面。

```text
GET/HEAD /__openclaw__/canvas...
```

返回 Canvas host 的静态资源，比如页面、JS、CSS、图片等。

```text
WS /__openclaw__/ws
```

这是 WebSocket，不是普通 HTTP 接口。用于 Canvas live reload 或 Canvas host 的实时通信。

插件 / UI：

```text
/api/channels/mattermost/command
/api/channels/mattermost/*
```

Mattermost 插件/频道回调入口。比如 Mattermost slash command 发到 OpenClaw gateway，由 gateway 转成内部消息或操作。

```text
{controlUiBasePath}...
```

Control UI 的页面和静态资源入口。比如 Web 控制台 SPA。

```text
registered plugin HTTP routes
```

插件注册的 HTTP 路由。不同插件可以扩展自己的 webhook、OAuth callback、控制接口等。

```text
Slack plugin HTTP routes
```

Slack 插件相关 HTTP 入口。通常用于 Slack events、commands、interactions、OAuth callback 等。

一句话总结：这些 HTTP 接口里，`/v1/*` 是给 OpenAI 兼容客户端用的，`/sessions/*` 和 `/tools/*` 是给 OpenClaw HTTP 控制/工具调用用的，`hooks` 是给外部系统触发 gateway 用的，`canvas/a2ui/controlUi` 是页面和静态资源，`Mattermost/Slack/plugin routes` 是频道插件入口。


`/health` 和 `/healthz` 在 OpenClaw 里基本是同一个东西：**健康检查别名**。

它们都映射到同一种状态：

```ts
const GATEWAY_PROBE_STATUS_BY_PATH = new Map([
  ["/health", "live"],
  ["/healthz", "live"],
  ["/ready", "ready"],
  ["/readyz", "ready"],
]);
```

所以：

```text
GET /health
GET /healthz
```

都会返回 gateway 是否“活着”。

区别主要是命名习惯：

```text
/health   更通用、可读性强
/healthz  Kubernetes / Google SRE 风格常见命名
```

`z` 没有特殊业务含义，更多是历史/约定。一些基础设施默认喜欢用 `/healthz`、`/readyz`。

同理：

```text
GET /ready
GET /readyz
```

也是同一个 readiness 检查的两个别名。

可以理解成：

```text
/health  == /healthz
/ready   == /readyz
```

语义区别在 `health` 和 `ready` 之间，而不是有没有 `z`。

`health/live`：进程还活着。

`ready`：进程已经准备好对外服务。

比如：

```text
/health  返回 200：gateway 进程没死
/ready   返回 503：gateway 活着，但依赖还没准备好
```

所以最终是四个路径，两组语义：

```text
存活检查:
GET/HEAD /health
GET/HEAD /healthz

就绪检查:
GET/HEAD /ready
GET/HEAD /readyz
```
