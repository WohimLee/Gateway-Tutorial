这个场景我会把它拆成两条链路：**音频上行** 和 **建议下行**。

你要的不是“音视频通话”，而是：

```text
现场有人说话
  -> 眼镜麦克风听到
  -> 服务端实时理解
  -> 生成应对建议
  -> 文本显示到眼镜
```

所以推荐通信形态是：

```text
眼镜 App
  -- 音频流上行 -->
Gateway / Realtime Service
  -- partial transcript / suggestions 下行 -->
眼镜 UI
```

**第一版我会用 WebSocket**

最务实的方案：

```text
一条 WebSocket 连接
  上行：binary audio frames
  下行：JSON events
```

例如连接后先发：

```json
{
  "type": "start",
  "sessionId": "meeting-123",
  "audio": {
    "encoding": "pcm_s16le",
    "sampleRate": 16000,
    "channels": 1,
    "frameMs": 20
  }
}
```

然后眼镜持续发 binary audio frame：

```text
20ms PCM16 mono
16kHz * 0.02 * 2 = 640 bytes
```

服务端返回 JSON：

```json
{
  "type": "transcript.partial",
  "speaker": "speaker_2",
  "text": "I’m worried this timeline is too tight",
  "startMs": 12400,
  "endMs": 15120
}
```

再返回建议：

```json
{
  "type": "suggestion",
  "priority": "high",
  "text": "可以先认可对方担忧，再问最影响进度的风险点是什么。",
  "reason": "对方表达了进度风险，需要降低防御感并收集约束。"
}
```

眼镜只显示 `text`，不要把 `reason` 都塞到 UI 上。

**服务端管线**

```text
Audio Ingest
  -> VAD
  -> Streaming ASR
  -> diarization / wearer-vs-other
  -> utterance buffer
  -> intent/emotion/tension detection
  -> response suggestion model
  -> display policy
  -> push to glasses
```

关键不是每句话都回复，而是做一个 **suggestion policy**，否则眼镜上会刷屏。比如只在这些情况给提示：

- 对方问了明确问题
- 对方表达反对/担忧
- 出现谈判/冲突/不确定
- 佩戴者沉默超过 N 秒
- 需要澄清、复述、追问
- 会议目标相关内容出现

**什么时候用 WebRTC**

如果你只上行一路眼镜麦克风，WebSocket 足够做第一版。WebRTC 更适合这些情况：

- 弱网下仍要稳定低延迟
- 音频质量非常重要
- 要用 Opus 压缩省带宽
- 要浏览器/移动端统一媒体栈
- 后续要双向语音交互
- 要接 LiveKit 这类实时房间/agent 框架

但即使用 WebRTC，**建议下行仍然通常走 data channel 或普通 WebSocket JSON**。媒体和控制/文本最好分开：

```text
WebRTC audio track 上行
WebSocket / WebRTC data channel 下行 suggestions
```

**我会推荐的落地路线**

第一阶段：

```text
眼镜 -> WebSocket binary PCM
服务端 -> WebSocket JSON suggestions
```

优点是最快、最可控、调试简单。

第二阶段：

```text
眼镜 -> WebRTC Opus audio via LiveKit
服务端 Agent 订阅 audio track
Agent -> WebSocket/DataChannel 推送 suggestions
```

优点是实时媒体体验更强，但系统复杂度高很多。

**最低可用延迟目标**

建议这样设计：

- 音频帧：20ms 或 40ms
- VAD endpoint：300-800ms
- partial transcript：持续返回
- suggestion：在完整语义片段后 500-1500ms 内出现
- UI：只显示 1-2 条候选，不滚动刷屏

一句话：**这个产品形态第一版最适合 WebSocket 双向连接：上行音频帧，下行转写和建议。WebRTC 是后续优化实时媒体传输质量的选择，不是第一天必须上。**
