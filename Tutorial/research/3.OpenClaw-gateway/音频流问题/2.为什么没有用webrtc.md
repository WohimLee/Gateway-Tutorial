不一定。**如果目标是浏览器/移动端之间做双向低延迟实时通话，WebRTC 通常更好；但 OpenClaw 现在这类 gateway/电话/agent 音频场景，用 WebSocket 反而更简单、更可控。**

主要原因大概是这几个。

**1. 现有音频入口本来就是 Twilio Media Streams**

`voice-call` 的实时音频来自 Twilio Media Streams。Twilio 给你的就是 WebSocket JSON frame，里面放 base64 的 μ-law 8kHz 音频。OpenClaw 只需要接 `/voice/stream`，再把音频转给 STT。

如果中间强行换成 WebRTC，反而要多做：

- Twilio WS → RTP/WebRTC 的桥
- SDP/ICE 信令
- jitter buffer
- 编解码转码
- NAT/TURN 处理
- 连接状态恢复

收益不大，复杂度飙升。

**2. Gateway 更像控制面，不是媒体服务器**

OpenClaw gateway 的主职责是：

- 节点/客户端连接
- RPC
- 权限和配对
- agent 事件
- 工具调用
- HTTP/OpenAI-compatible API

它不是 SFU/MCU，也不是实时音视频网关。WebRTC 会把 gateway 推向“媒体服务器”方向，需要维护 ICE、DTLS、SRTP、RTP、codec negotiation、TURN 等一整套东西。

WebSocket 音频流虽然没 WebRTC 那么媒体专用，但对 **单路语音 → STT → TTS 回复** 很够用。

**3. OpenAI Realtime STT 这里也是 WebSocket**

当前代码接 Twilio 的 μ-law 8kHz，然后转发到 OpenAI Realtime STT WebSocket，配置的是 `g711_ulaw`。也就是说链路天然是：

```text
Twilio WebSocket -> OpenClaw WebSocket handler -> OpenAI Realtime WebSocket
```

全链路不需要 WebRTC。用 WebRTC 反而要在 OpenClaw 里转换一次。

**4. WebSocket 更容易和 agent/RPC/日志/权限模型融合**

OpenClaw 已经有 gateway WS 协议、scope、auth、device token、事件广播。音频流用 WebSocket，工程上可以沿用很多模型：

- 明确的 JSON event
- 易记录、易测试
- 易做 backpressure 限制
- 易做鉴权 token
- 易在 Node 里跑
- 易和电话 provider webhook 共用 HTTP server

WebRTC 的二进制媒体流更适合“实时媒体”，但不如 WebSocket 容易纳入 agent 平台控制逻辑。

**5. WebRTC 的低延迟优势在这里未必是瓶颈**

WebRTC 优势是：

- UDP
- jitter buffer
- Opus
- NAT 穿透
- 自适应码率
- 双向实时媒体优化

但电话链路本身是 8kHz μ-law，STT/VAD/LLM/TTS 才是主要延迟来源。对语音 agent 来说，WebRTC 少掉的几十毫秒可能不如 STT/TTS、模型响应、电话网络延迟影响大。

所以 WebSocket 是一种很务实的选择。

**什么时候 WebRTC 会更好？**

如果 OpenClaw 要做这些，我会考虑 WebRTC：

- 浏览器直接和 gateway 实时语音对话
- 移动端把麦克风流实时传到 gateway，而不是本机 STT
- 双向低延迟 interrupt/barge-in 要做到很丝滑
- 多人语音房间、会议、Discord-like voice
- 要用 Opus、回声消除、噪声抑制、自动增益这些实时音频能力
- gateway 要接 LiveKit、mediasoup、Janus、Daily、Twilio Voice SDK 这类 RTC 平台

但这时我也不建议 OpenClaw 自己裸写 WebRTC。更现实的是接一个 RTC 服务：

```text
Browser/Mobile --WebRTC--> LiveKit/mediasoup/Twilio Voice SDK
                         -> OpenClaw agent bridge
                         -> STT/LLM/TTS
```

**结论**

WebRTC 在“实时人声媒体传输”上确实更专业；但 OpenClaw 当前的音频流是电话 provider + STT/TTS agent 管线，WebSocket 更贴合现有接口，也更容易维护。

所以不是 WebRTC 不好，而是这里 **WebSocket 是成本/收益更好的选择**。
