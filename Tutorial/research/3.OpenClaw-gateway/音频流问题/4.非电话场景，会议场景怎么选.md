如果是“会议里的某个麦克风/房间麦克风”，并且要做 **ASR + diarization + 情感分类**，我会按目标分两种方案：

**最佳架构：WebRTC/会议媒体层 + 专业语音分析服务**

如果你能控制会议接入，推荐：

```text
麦克风 / 会议客户端
  -> WebRTC 房间，比如 LiveKit / mediasoup / Janus
  -> 音频轨道采集
  -> ASR + diarization
  -> 情感/意图/摘要/行动项分析
```

这里 **WebRTC 比 WebSocket 音频更合适**，因为会议音频天然是实时媒体场景，需要：

- 低延迟
- 回声消除
- 噪声抑制
- 自动增益
- jitter buffer
- 丢包恢复
- 多人音轨管理
- mute/unmute、active speaker、音量检测

如果每个参会人有独立音轨，最好不要先混音。**独立音轨比混合后的单路房间麦克风更利于 speaker attribution**。能拿到“谁的 track 在说话”，diarization 压力会小很多。

**如果只有一个房间麦克风**

那就是：

```text
单路混合音频
  -> VAD / 降噪 / 分段
  -> streaming ASR with diarization
  -> 按 utterance 做 sentiment/emotion/intent
```

这种情况下 diarization 会难很多，尤其是：

- 多人重叠说话
- 远场麦克风
- 回声
- 桌面噪声
- 说话人声音相近
- 人数超过 4-6 个

这时建议用支持 **streaming diarization** 的 ASR 服务，而不是自己从零做。

**服务选择**

- **LiveKit**：适合做会议/WebRTC 音频入口和 voice agent 管线。它的 Agents STT 文档里也明确有 diarization / primary speaker 相关适配，并列出 AssemblyAI、Deepgram、Speechmatics、Soniox 等 diarization provider。  
  https://docs.livekit.io/agents/models/stt/

- **Deepgram**：适合低延迟 streaming ASR，常用于实时转写、diarization、会话分析。情感分类通常放在 transcript/utterance 层做。  
  https://deepgram.com/learn/real-time-sentiment-analysis-streaming-audio

- **AssemblyAI**：如果你更看重“ASR + speaker labels + sentiment analysis”一体化，AssemblyAI 很顺，尤其是异步/录音后分析。它支持把 sentiment result 关联 speaker label。  
  https://www.assemblyai.com/docs/speech-understanding/sentiment-analysis

- **Amazon Transcribe**：企业/AWS 环境友好，支持 streaming speaker diarization。AWS 文档说可在 streaming transcription 里开启 speaker partitioning。  
  https://docs.aws.amazon.com/transcribe/latest/dg/diarization.html

**我的实际建议**

如果你是在做一个会议助手/会议分析系统：

```text
LiveKit 做会议音频入口
Deepgram 或 AssemblyAI 做 ASR + diarization
LLM 做情感/风险/行动项/摘要
结构化存储每个 utterance
```

数据结构大概长这样：

```json
{
  "speaker": "speaker_2",
  "start_ms": 18420,
  "end_ms": 23110,
  "text": "I don't think we can ship this by Friday.",
  "sentiment": "negative",
  "emotion": "concerned",
  "confidence": 0.87
}
```

关键点：**ASR/diarization 处理音频，情感分类最好处理“带 speaker + timestamp 的 utterance 文本”**，不要直接对裸音频流硬做全套，除非你有很强的语音模型需求。

一句话：电话场景用 Twilio Media Streams/WebSocket 很合理；**会议麦克风场景，我会用 WebRTC/LiveKit 做音频入口，再接专业 ASR diarization 服务。**
