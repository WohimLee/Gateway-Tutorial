能做到“每个参会人独立 audio track”的前提是：**每个人本来就有独立采集源**。比如每个人用自己的手机/电脑进 LiveKit room，每个人发布一条 audio track。

但你这个场景不是这样。你说的是：

```text
现场会议
  -> 戴眼镜的人
  -> 眼镜上的一个麦克风
  -> 采到整个房间的混合声音
```

那就 **做不到真正的每人独立 audio track**。因为声音在空气里已经混在一起了，单麦克风收到的是一个混合波形。后端只能做 **speaker diarization**，也就是从混合音频里猜“这段是谁说的”。

这两者差别很大：

| 场景 | 能力 |
|---|---|
| 每个人一个麦克风 | 天然知道谁在说话，speaker attribution 很准 |
| 一个房间麦克风 | 只能从声音特征推断 speaker，准确率受环境影响很大 |

对智能眼镜场景，我会这样看：

**1. WebRTC 不是为了 diarization，而是为了稳定传输实时音频**

眼镜麦克风只有一路音频。WebRTC 可以帮你把这一路音频低延迟传到服务器，但它不会神奇地分离每个人。

你需要的是：

```text
眼镜麦克风
  -> 音频采集/降噪/VAD
  -> WebRTC 或 WebSocket 上传
  -> ASR
  -> diarization
  -> 语义理解/摘要/情感/行动项
```

**2. 如果只是上传一路麦克风，WebSocket 也可以**

如果你控制眼镜端软件，并且目标是 ASR/会议记录，不是实时双向通话，第一版用 WebSocket 音频流完全可以：

```text
PCM16 16kHz mono
20ms/40ms frame
binary WebSocket
server -> ASR diarization
```

WebRTC 更适合弱网、实时交互、回声消除、长连接媒体传输。WebSocket 更简单，调试更快。

**3. 单麦克风 diarization 的现实上限**

单眼镜麦克风在现场会议里会遇到：

- 远处的人声音小
- 多人重叠说话
- 投影/空调/键盘噪声
- 眼镜佩戴者自己的声音特别近
- 转头会改变声场
- 同性别/相似音色的人难区分
- 房间混响

所以 diarization 可以做，但不要期待像“每个人独立麦克风”那么准。

**4. 智能眼镜更靠谱的增强方式**

如果硬件允许，优先考虑：

- 双麦/多麦阵列，而不是单麦
- beamforming，尽量朝向说话方向增强
- VAD，先分出有效语音段
- 近场/远场分类，区分佩戴者本人和外部说话人
- 佩戴者 voice profile，用来识别“我”
- 会议开始前让系统做一次 speaker enrollment，比如每个人说一句话
- 如果可以，用手机/会议室设备辅助采音

**5. 推荐技术路线**

对你们智能眼镜，我会建议：

第一阶段，简单可靠：

```text
眼镜端：
  采集 PCM16 16kHz/24kHz mono
  本地 VAD 可选
  WebSocket binary 分帧上传

服务端：
  streaming ASR
  diarization
  utterance 分段
  LLM 做摘要/行动项/情感分类
```

第二阶段，提升实时性和抗弱网：

```text
眼镜端：
  WebRTC audio track
  Opus 编码
  AEC/NS/AGC/抖动处理

服务端：
  LiveKit/mediasoup 接入
  agent 订阅音频 track
  ASR + diarization
```

第三阶段，提升会议说话人识别：

```text
多麦阵列 / beamforming
speaker enrollment
wearer-vs-other 分类
声纹聚类 + 人名映射
和日历/会议参会人名单结合
```

一句话：**你这个场景里，“独立 audio track”做不到，因为物理采集源只有一个；真正要攻的是单麦/近远场会议音频的 ASR + diarization + wearer detection。**
