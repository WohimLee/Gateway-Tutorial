你说的应该是 **LiveKit**。

它可以理解成：**一个开源的实时音视频基础设施 + AI 语音 Agent 框架**。更白话一点，它像是“你自己可控的 Zoom/Discord 语音房间底座”，但给开发者提供 SDK、服务端、WebRTC 房间、音频轨道、视频轨道、数据通道、录制/导出、Agent 接入这些能力。

它通常解决的是这个问题：

```text
浏览器 / iOS / Android / 桌面端
  -> WebRTC 进入 LiveKit room
  -> 后端 Agent 也作为 participant 加入 room
  -> Agent 读取音频/视频轨道
  -> STT / LLM / TTS
  -> 把语音/文本/视频结果发回 room
```

和你前面问的 WebSocket 音频相比，LiveKit 帮你处理 WebRTC 里很麻烦的部分：

- 房间管理
- 参会者管理
- 音频/视频 track 发布和订阅
- NAT 穿透
- ICE/TURN/STUN
- SFU 转发
- 网络质量自适应
- mute/unmute
- active speaker
- 音频/视频同步
- 跨 Web、iOS、Android、Flutter、后端 SDK

它的核心组件大概是：

| 组件 | 作用 |
|---|---|
| LiveKit Server | WebRTC SFU，负责实时媒体转发 |
| LiveKit SDKs | Web/iOS/Android/Flutter/后端接入房间 |
| LiveKit Agents | Python/Node Agent 框架，做 STT-LLM-TTS 管线 |
| LiveKit Cloud | 官方托管版 |
| Self-host | 自己部署开源 LiveKit server |

如果你的场景是“会议里的某个麦克风，要实时 ASR、说话人识别、情感分析”，LiveKit 的价值是：**先把会议音频变成可管理的实时音频轨道**。然后你的 Agent 或后端订阅这些 track，再送给 Deepgram、AssemblyAI、Whisper、OpenAI、AWS Transcribe 等服务。

典型架构：

```text
会议客户端
  -> LiveKit Room
  -> Agent 订阅 audio track
  -> ASR / diarization
  -> 情感分类 / 摘要 / 行动项
  -> 返回字幕、事件、结构化数据
```

如果你只有一个麦克风，LiveKit 负责稳定传音频；diarization 还是交给 ASR 服务。如果每个参会人各自进房间，LiveKit 能保留每个人的独立音频 track，这比混成一路再做 diarization 好很多。

一句话：**Twilio Media Streams 是电话音频接入方案；LiveKit 是会议/实时音视频/WebRTC 接入方案。**
