有些场景 **SSE 确实解决不了，或者会很别扭**，这时就该上 **WebSocket**。

先说结论：

**只要你们的客户端需要“持续、低延迟、主动地往服务端发消息”，SSE 就不够了。**
因为 **SSE 本质上只有一个方向：服务器 → 客户端**。客户端要发东西，仍然得另外再发 HTTP 请求。

---

## SSE 能搞定的场景

SSE 很适合这种：

* 客户端先发起一次请求
* 然后主要是服务器持续往下推结果
* 客户端很少中途发控制消息

例如：

* 大模型文字流式输出
* 实时字幕下发
* 任务进度通知
* 订单状态更新
* 告警推送

这类场景里，通信主方向是**下行**，所以 SSE 很顺。

---

## 哪些情况 SSE 不行，或者明显不如 WebSocket

### 1. 需要真正的双向实时交互

比如你们眼镜端和云端要不断来回发消息：

* 用户开始说话
* 客户端持续上传音频分片
* 服务端持续回 ASR 中间结果
* 客户端发打断
* 服务端立刻停 TTS
* 客户端再发新一轮输入
* 服务端继续回状态

这类是**双向连续交互**。
SSE 只能负责“服务端往下推”，客户端上行还得靠额外 HTTP。这样链路会变成：

* 上行：很多个 HTTP 请求
* 下行：1 条 SSE

能做，但会很别扭，状态管理也复杂。
这时 WebSocket 一条连接里双向收发更自然。

---

### 2. 客户端需要频繁发小消息

例如：

* 心跳
* 设备状态更新
* 电量变化
* 网络质量上报
* 传感器事件
* 用户点击/手势
* 会话中的 cancel / interrupt / ack

如果这些消息很多，SSE 就不合适，因为它根本不负责客户端上行。
你只能不停发 HTTP 请求补上行，这会把系统做成“半拉子实时”。

**这种高频双向小消息，基本就该 WebSocket。**

---

### 3. 需要低延迟“打断”

这个对智能助手特别关键。

比如：

* 云端正在播报
* 用户突然说“停”
* 眼镜端要立刻告诉服务端停止生成/停止 TTS
* 服务端还要马上确认已停止

SSE 下，服务端到客户端没问题；
但客户端到服务端这个“停”得额外发 HTTP。

如果你们对打断延迟要求高，SSE 方案就容易变得笨重。
WebSocket 在这种“随时发控制指令”的场景更合适。

---

### 4. 需要一个统一的实时状态机

如果你们有下面这种协议需求：

* request
* response
* event
* ack
* error
* cancel
* timeout
* sequence
* session state

那你们已经不是单纯“服务器推消息”了，而是在做一个**实时会话协议**。
这种协议更适合放在 WebSocket 上。

SSE 更像“广播流”或“事件流”，
WebSocket 更像“实时会话通道”。

---

### 5. 需要客户端主动订阅/退订/切换流

例如：

* 切换字幕语言
* 切换识别模式
* 切换视频质量
* 打开/关闭视觉流
* 切换不同会话房间
* 订阅不同类型事件

如果这些动作会在一个会话里频繁发生，WebSocket 很顺手。
SSE 也不是完全不能做，但往往要靠：

* 断开旧 SSE
* 再发新的 HTTP
* 重建订阅上下文

体验和实现都更绕。

---

### 6. 你们在传“持续上行数据”

最典型的是：

* 连续音频 chunk 上行
* 连续传感器数据上行
* 持续帧/截图上行

SSE 完全不能承担这个上行角色。
它只能下行。

严格说，这种场景不是“WebSocket 非得不可”，因为也可能用别的实时媒体方案；但**在 SSE 和 WebSocket 二选一时，只能选 WebSocket**。

---

## 什么时候 SSE 足够，没必要 WebSocket

下面这些我会优先 SSE：

### 1. 服务端单向推送文本

比如：

* LLM token 流
* 翻译字幕
* 实时摘要
* 任务进度

### 2. 客户端上行很少

比如只在开始时发一个请求，后面主要等结果。

### 3. 浏览器/网关环境更偏 HTTP 生态

SSE 通常比 WebSocket 更容易接入传统 HTTP 基础设施。

### 4. 你们不想维护复杂连接状态

SSE 更简单，适合“推送为主”的链路。

---

## 一句判断标准

你可以这样分：

### 用 SSE

当通信模式是：

**“我发起一次请求，然后你持续告诉我结果。”**

### 用 WebSocket

当通信模式是：

**“我们在一个会话里要不断双向对话，双方都可能随时主动发消息。”**

---

## 放到你们智能眼镜场景里

### SSE 能做的

* 云端往眼镜推：

  * LLM 文本流
  * 实时字幕
  * 任务进度
  * 状态通知

### SSE 做不好的

* 眼镜持续上传音频 chunk
* 眼镜随时发 interrupt / cancel
* 设备持续上报状态
* 会话过程中频繁切模式
* 双向实时控制

这些更适合 WebSocket。

---

## 你问“什么情况下非得走 WebSocket”

更严谨地说，不是“全世界只有 WebSocket 能做”，而是：

**在“只比较 SSE 和 WebSocket”时，只要你需要持续双向实时通信，SSE 就解决不了，必须上 WebSocket。**

核心边界就一句：

**SSE 不是双向协议。**

---

## 给你一个最短版结论

下面这些，SSE 基本不够：

* 客户端频繁上行消息
* 需要低延迟打断
* 双向实时会话
* 持续上传音频/传感器数据
* 一个连接里管理复杂状态机

这时该用 **WebSocket**。

而如果只是：

* 服务端持续往下推文本/事件
* 客户端很少中途发东西

那 **SSE 就够了**。
