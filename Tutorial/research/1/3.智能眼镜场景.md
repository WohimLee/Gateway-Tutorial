如果你们是**智能眼镜 + 云端智能助手**，我建议不要只选一种，而是按链路拆开：

**最实用的组合：**

* **HTTP**：登录、鉴权、配置拉取、图片上传、短视频上传、历史记录查询
* **SSE / Streaming HTTP**：助手文本回复、状态事件下发、逐步返回识别/推理结果
* **WebSocket**：设备控制信令、会话状态同步、低频双向事件
* **WebRTC**：实时语音，和需要低时延的视频上行/双向音视频

这是因为：
HTTP 是标准请求-响应；SSE 是服务器到客户端的持续推送；WebSocket 是双向交互；WebRTC 专门就是为实时音视频和低时延数据设计的。([MDN Web Docs][1])

### 先给结论：别用一个协议硬扛全部场景

对于智能眼镜，通信通常分成 4 类：

1. **控制类**
   例如开始录音、停止录音、切换模式、心跳、网络状态、打断助手、更新字幕状态。
   这类消息小、频繁、双向，适合 **WebSocket**。WebSocket 允许连接建立后持续双向通信，不需要反复轮询。([MDN Web Docs][1])

2. **请求类**
   例如上传一张图让云端分析、上传一段短视频做总结、获取用户配置、拉取会话历史。
   这类天然适合 **HTTP**，因为就是一次请求对应一次结果，易做鉴权、重试、CDN、网关治理。([MDN Web Docs][1])

3. **结果流式返回**
   例如 ASR 中间结果、LLM token 流式输出、任务进度、翻译字幕持续下发。
   这类适合 **SSE 或流式 HTTP**。SSE 本质上是 HTTP 上的服务器持续推送，浏览器和很多客户端实现都比较简单。([MDN Web Docs][2])

4. **实时媒体**
   例如连续麦克风上行、对讲式语音助手、第一视角实时视频理解、远程协助。
   这类优先 **WebRTC**，因为它就是为实时音视频和低时延传输设计的，并且配套了 ICE / STUN / TURN 等穿透和连接机制。([MDN Web Docs][3])

## 按你们的业务，我会这样选

### 1）语音助手主链路：优先 WebRTC

如果你们要做的是下面这种体验：

* 用户说话时边传边识别
* 云端边理解边返回中间结果
* 支持打断
* 需要低时延 TTS 回放
* 网络抖动下还要尽量稳

那主语音链路优先 **WebRTC**。
原因不是“它更潮”，而是它对实时媒体更合适：音视频采集、传输、抖动缓冲、丢包处理、加密与连接协商，本来就是它的主场。([MDN Web Docs][3])

**适合你们的方式：**

* 眼镜麦克风上行：WebRTC 音频流
* 云端返回 TTS 音频：WebRTC 下行音频流
* 会话控制信令：WebSocket
* 文本中间结果：SSE 或 WebSocket

**不建议**把“连续实时语音”主链路建立在普通 HTTP 上传分片上，除非你们接受更高时延、较弱互动性，只做“按住说一句、等结果”的弱实时助手。

### 2）图片理解：HTTP 上传

如果是“拍一张图 → 云端分析 → 返回答案”，首选 **HTTP**。
因为图片通常是离散请求，不需要维持持续媒体会话；HTTP 更容易做鉴权、压缩、缓存、对象存储、失败重试和链路观测。

**典型流：**

* 拍照
* HTTP 上传到对象存储或推理网关
* 返回任务 ID 或直接返回结果
* 若结果耗时稍长，再用 SSE/WS 回推状态

这个场景没必要为一张静态图片开 WebRTC。

### 3）短视频片段分析：默认 HTTP，实时远程协助才上 WebRTC

这里要分两种：

#### A. “录 5 秒/10 秒再分析”

比如：

* 运动回放总结
* 通勤路况回顾
* 佩戴者视角行为摘要

这类建议 **HTTP 分片/直传上传**。
因为它不是实时交互，HTTP 成本低、架构简单。

#### B. “实时看我看到的画面”

比如：

* 远程专家协助
* 实时导航理解
* 边走边识别环境
* 边拍边问助手“我现在看到什么”

这类才建议 **WebRTC 视频上行**。
因为这里低时延比吞吐更重要，视频是连续媒体流，WebRTC 更合适。([MDN Web Docs][3])

### 4）助手文本输出：SSE 很合适

如果云端助手要把这些东西持续推给眼镜：

* LLM token 流式输出
* 实时字幕
* 翻译中间结果
* 任务进度
* 提示卡片状态变化

我会优先考虑 **SSE**，尤其是“服务器持续往设备推文本/事件”这种模式。
SSE 基于 HTTP，简单、易调试，天然适合下行事件流。([MDN Web Docs][2])

如果你们客户端环境不是浏览器，而是 Android / Linux / 嵌入式 App，也一样能实现 SSE；只是工程上有时团队会因为统一栈而把这类事件也放进 WebSocket。

### 5）控制与信令：WebSocket

下面这些我会放 WebSocket：

* 设备在线状态
* 心跳
* 会话创建/结束
* 打断助手
* 音视频会话协商的控制消息
* 字幕显示开关、模式切换
* 服务器主动下发轻量控制事件

原因很简单：这些都是**小消息、双向、低延迟、持续连接**。
WebSocket 非常适合这种控制面。([MDN Web Docs][1])

## 给你一个推荐架构

### 方案 A：大多数智能眼镜助手的稳妥方案

适合“语音助手为主，图片/短视频为辅”

* **HTTP**

  * 登录、刷新 token
  * 上传图片
  * 上传短视频片段
  * 拉配置、拉历史
* **SSE**

  * LLM 文本流
  * ASR 中间结果下发
  * 任务进度通知
* **WebSocket**

  * 设备心跳
  * 会话状态
  * 控制消息
  * 打断 / 取消 / 模式切换
* **WebRTC**

  * 实时麦克风上行
  * TTS 音频回传
  * 可选实时视频上行

这是我最推荐的。

### 方案 B：先快速落地 MVP

适合先做出能用的版本

* **HTTP**：图片、短视频、配置
* **WebSocket**：所有控制 + 文本流 + 中间结果
* **暂时不做 WebRTC**
* 语音先做“按住说一句 → 上传音频片段 → 返回结果”

这个方案开发更快，但体验上限会低一些，特别是在：

* 打断
* 半双工/全双工语音
* 连续对话
* 低时延回声体验

### 方案 C：高实时远程协助版

适合“远程专家 + 第一视角”场景

* **WebRTC**

  * 实时音频双向
  * 实时视频上行
  * 必要时数据通道传少量控制数据
* **WebSocket**

  * 业务控制信令
  * 会话状态
* **HTTP**

  * 截图、录像回传、日志、配置
* **SSE**

  * AI 文本解释/总结流

## 我不建议的几种做法

### 1. 全部都走 WebSocket

看起来统一，实际上容易把控制面和媒体面搅在一起。
WebSocket 很适合事件和信令，但不等于它是实时音视频的最佳主链路。对连续媒体，WebRTC 更对口。([MDN Web Docs][1])

### 2. 全部都走 HTTP 轮询

这个会把实时体验做坏：

* 延迟高
* 功耗高
* 网络浪费
* 打断能力差

### 3. 让图片/短视频分析也强行走 WebRTC

除非你们就是在做持续实时视觉理解，否则上传离散文件时，HTTP 更简单、更稳。

## 按业务目标来选

### 如果你们核心是“语音助手”

选：

* 语音：**WebRTC**
* 文字流：**SSE**
* 控制：**WebSocket**
* 图片/短视频：**HTTP**

### 如果你们核心是“拍照问答”

选：

* 图片：**HTTP**
* 结果流：**SSE**
* 控制：**WebSocket**
* 可先不做 WebRTC

### 如果你们核心是“第一视角实时协助”

选：

* 音视频：**WebRTC**
* 控制：**WebSocket**
* 摘要/文字：**SSE**
* 存档上传：**HTTP**

## 你们这种设备还有两个现实约束

### 功耗

智能眼镜比手机更敏感。
持续保持高负载视频流会显著影响续航，所以不要默认常开视频上行。更合理的是：

* 常驻低成本控制通道
* 需要时才升起实时媒体通道
* 图片优先于持续视频

这是架构上非常重要的一点。

### 弱网切换

眼镜经常经历：

* 手机热点
* Wi-Fi
* 室外蜂窝
* 进电梯/地库

所以你们要把链路分层：

* 控制面掉线要能快速恢复
* 媒体面要能单独重建
* 文件上传要支持断点/重试
* 事件流要支持幂等和重连续传

## 我的最终建议

如果让我替你们拍板，我会这样定：

**第一阶段可商用版本**

* **HTTP**：图片、视频片段、登录配置
* **WebSocket**：设备控制、会话状态、打断
* **SSE**：字幕、ASR/LLM/TTS 状态、文本流
* **WebRTC**：只用于实时语音

**第二阶段高阶版本**

* 在上面基础上，把“实时第一视角视频理解/远程协助”也接入 **WebRTC 视频上行**

这样做的好处是：

* 架构清晰
* 媒体和控制解耦
* 易于从 MVP 演进
* 不会过早把所有复杂度都压到一个协议上

一句话总结：

**图片/短视频走 HTTP，文本流走 SSE，控制走 WebSocket，实时语音/视频走 WebRTC。**
