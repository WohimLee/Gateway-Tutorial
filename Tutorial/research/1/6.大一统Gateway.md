# 二、为什么“统一 Gateway”是主流方案？

你前面问的所有问题，本质上都在指向一件事：

👉 **不同通信方式适合不同数据类型**

所以合理的做法不是“选一个”，而是：

> **让 Gateway 帮你做协议分发和统一治理**

---

## 它解决了什么问题？

### 1️⃣ 对客户端统一入口

设备端只需要记一个域名：

```text
wss://api.xxx.com
https://api.xxx.com
```

不用关心后面是：

* ASR 集群
* LLM 集群
* 视频服务
* 对象存储

---

### 2️⃣ 协议能力集中

一个 Gateway 可以同时支持：

* HTTP（REST / 上传下载）
* SSE（事件流）
* WebSocket（双向）
* WebRTC（信令 + ICE 辅助）

👉 客户端按需选择

---

### 3️⃣ 统一治理（这是最关键的价值）

所有流量在这里做：

* 鉴权（token / device id）
* 限流
* 日志 / tracing
* 灰度发布
* A/B test
* 路由（按用户 / 场景分流）

👉 不用每个服务都自己搞一套

---

# 三、在你们智能眼镜场景里的“标准拆法”

我给你一个比较“工业级”的拆法（很接近真实公司）

---

## 1️⃣ HTTP 通道（文件 & 请求）

**Gateway 负责：**

* 上传图片
* 上传短视频
* 获取配置
* 拉历史

```text
POST /vision/upload
GET  /user/config
```

👉 后面转：

* 对象存储
* 视觉推理服务

---

## 2️⃣ SSE 通道（下行事件流）

```text
GET /assistant/stream
```

用于：

* LLM token 流
* 字幕
* 任务进度

👉 Gateway：

* 建立长连接
* 订阅后端事件（Kafka / 内部流）
* 转发给客户端

---

## 3️⃣ WebSocket 通道（控制 & 会话）

```text
wss://api.xxx.com/ws
```

用于：

* 会话管理
* interrupt / cancel
* 设备状态
* 控制信令

👉 Gateway 做：

* 连接管理（百万连接）
* session 绑定
* 路由到对应服务

---

## 4️⃣ WebRTC（实时媒体）

这里有个关键点：

👉 **WebRTC 通常不“完全走 Gateway”**

而是：

* Gateway 负责：

  * signaling（信令）
  * 鉴权
* 媒体流：

  * 直接走 SFU / 媒体服务器

```text
设备 → WebRTC → 媒体节点（SFU）
             ↘ 信令走 Gateway
```

👉 这是因为：

* 媒体流太重
* Gateway 扛不住也不适合做媒体转发

---

# 四、统一 Gateway 的关键设计点（重点）

## 1️⃣ 多协议支持

一个 Gateway 要支持：

* HTTP 1.1 / 2
* WebSocket upgrade
* SSE（chunked response）
* WebRTC signaling（通常也是 HTTP/WS）

---

## 2️⃣ 连接管理能力（非常难的点）

特别是 WebSocket：

* 长连接数（10万 / 100万级）
* 心跳检测
* 断线重连
* session 粘性（sticky routing）

---

## 3️⃣ 路由策略

比如：

```text
/ws → 会话服务
/assistant/stream → LLM
/vision/upload → 视觉服务
```

甚至：

```text
按用户 → 不同模型
按设备 → 不同区域
```

---

## 4️⃣ 状态管理

WebSocket 带来一个问题：

👉 Gateway 是否要保存 session 状态？

两种方案：

### 无状态（推荐）

* Gateway 只做转发
* session 在后端

### 有状态

* Gateway 持有连接上下文
* 更复杂，但延迟更低

---

## 5️⃣ 扩展性（很关键）

HTTP 很容易扩展（无状态）
WebSocket / WebRTC：

👉 要考虑：

* 连接迁移
* sticky session
* 分片（sharding）
